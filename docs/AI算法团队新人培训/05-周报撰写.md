# 第五章：周报撰写规范

> 好的周报让上级在 3 分钟内获得决策所需的全部信息。坏的周报是 10 分钟读完什么都不知道。

---

## 1. 周报的本质与目的

### 规则

**周报不是"工作流水账"，是给上级的决策信息包。**

每份周报必须回答三个问题，且只需回答这三个问题：

| 问题 | 对应内容 |
|------|---------|
| 你这周产出了什么价值？ | 完成事项 + 量化结果 |
| 有什么需要我知道的风险？ | 问题与解决方案 |
| 下周你会做什么？ | 具体计划 + 优先级 |

**写周报之前，先问自己：** "如果上级只有3分钟看这份周报，他能做出哪些判断？"

---

### ❌ 反例（流水账式周报）

```markdown
本周工作总结：

周一：看了DETR的论文，了解了Transformer在检测中的应用。
      下午参加了组会，听了大家的分享。

周二：配置了实验环境，安装了各种依赖，遇到了一些版本冲突问题，
      后来解决了。下午开始写数据加载的代码。

周三：继续写代码，数据加载部分基本完成了。下午跑了一个实验，
      效果还可以。

周四：参加了项目例会。下午调参，跑了几个实验。

周五：整理代码，写了一些注释。下午写这个周报。

下周计划：继续推进实验，争取有更好的结果。
```

**逐条分析问题：**

- 全是过程描述（"写了代码"），没有结果（"代码实现了什么"）
- 没有任何量化数字
- 提到"解决了"版本冲突，但没说解决了什么，以后遇到同样问题还得重头来
- "跑了几个实验"——什么实验？结果是什么？
- 下周计划完全无法验收（"继续推进"是什么意思？）
- 读完这份周报，上级无法判断项目是否在正常轨道上

---

### ✅ 正例（价值导向式周报）

```markdown
**本周工作周报（2024-03-22，张三）**

**① 本周完成事项**
1. 完成 FPN Neck 模块开发并合并至 dev 分支（PR #23），已通过代码审查

   FPN + CBAM 注意力使小目标 Recall 从 61% 提升至 74%（+13%）
3. 完成数据预处理流水线重构，处理速度从 45 张/秒提升至 120 张/秒（+167%）

|------|-------------|-----|

| + FPN | 70% | 42 |

距目标 Recall 80% 还差 6%，本周里程碑 M2 已完成。

**③ 遇到的问题及解决方案**
- 问题：A100-1 节点 CUDA 版本冲突导致周三停工半天
  解决：切换至 A100-2 节点（CUDA 12.1），已恢复正常，不影响里程碑
- 风险：真实产线数据与测试集分布有偏差，初步验证 Recall 下降 8%
  计划：下周申请产线标注数据（500张），用于微调

**④ 下周计划**
1. 【优先级高】申请并获取 500 张产线标注数据（周一前完成申请）
2. 【优先级高】完成定制化 Mosaic 数据增强方案，验证能否补足 6% 的 Recall 缺口（截止周三）
3. 【优先级中】开始调研 TensorRT 推理优化方案，为 M3 部署做准备（截止周五）

**⑤ 需要支持的事项**
- 请帮忙协调标注团队，尽快审批 500 张产线数据标注申请
  预计标注成本：2人天，影响 M3 时间线
```

**读完这份周报，上级可以立刻判断：**

- 项目进展正常（M2 完成）✓
- 有一个已知风险（产线分布偏差），已有应对计划 ✓
- 需要帮助协调标注资源 ✓
- 下周有明确可验收的目标 ✓

## 2. 周报标准结构

### 规则

**固定五段式，每次按顺序写，写完对照 Checklist 检查。**

```
① 本周完成事项（结果 > 过程）
② 关键数据 / 实验结论（量化优先）
③ 遇到的问题及解决方案（必须写，哪怕没问题也写"无"）
④ 下周计划（具体任务 + 优先级 + 截止时间）
⑤ 需要支持的事项（如有）
```

**各段的写法要点：**

| 段落 | 要点 | 常见错误 |
|------|------|---------|
| ① 完成事项 | 每条以动词开头，说明完成了什么、产出是什么 | 写"研究了 X"而不是"完成了 X" |
| ② 关键数据 | 用表格呈现多组数据对比 | 只写文字描述，没有数字 |
| ③ 问题 | 写问题 + 已尝试的方案 + 当前状态 | 只写问题，不写解决方案 |
| ④ 下周计划 | 每条有截止时间和优先级 | 写"继续推进"等无法验收的词 |
| ⑤ 需要支持 | 写清楚需要什么、什么时候需要 | 这段不写，导致上级不知道你需要帮助 |

## 3. 措辞规范

### 规则

**动词的质量决定周报的质量。**

**高质量动词（产出感强）：**

```
完成 / 实现 / 验证 / 提升 / 解决 / 发现 / 落地 / 交付 /
发布 / 合并 / 对比 / 识别 / 优化 / 重构 / 调研 / 确认
```

**低质量动词（过程感强，无法验收）：**

```
做了 / 搞了 / 弄了 / 看了 / 研究了 / 了解了 /
跟进中 / 在推进 / 有进展 / 继续 / 努力
```

**量化公式：**

```
[高质量动词] + [具体对象] + [可度量结果]
```

---

### ❌ 反例

```
- 研究了一下 Transformer 的注意力机制
- 跑了几个实验，效果有提升
- 和导师讨论了一下方案
- 代码优化了一些
- 环境配置问题解决了
```

---

### ✅ 正例

```
- 精读 Deformable DETR 等3篇论文，整理对比笔记（见附件），
  识别出2个可迁移到当前任务的改进点（多尺度注意力 + 双分支解码器）

- 完成 FPN + CBAM 消融实验（4组），验证 FPN 贡献最大（Recall +9%），
  CBAM 进一步提升 4%，组合方案为当前最优

- 与导师确认了数据增强方案，决定放弃标准 Mosaic（小目标比例下降12%），
  改用定制化 Mosaic（下周实现并验证）

- 重构数据预处理流水线，将处理速度从 45 张/秒提升至 120 张/秒，
  代码已合并至 dev 分支（PR #21）

- 解决 A100-1 节点 CUDA 版本冲突问题，根本原因为 torch 2.x 需要
  CUDA 11.8+，解决方案：切换至 A100-2 节点（CUDA 12.1），
  已记录在 wiki/环境配置.md 中
```

**对比分析：**

| 版本 | 动词 | 对象 | 结果 | 可验收？ |
|------|------|------|------|---------|
| 研究了注意力机制 | 研究（弱）| 模糊 | 无 | ❌ |
| 精读3篇论文，识别2个改进点 | 精读/识别（强）| 具体 | 有 | ✅ |

---

### ✏️ Checklist

- [ ] 是否删除了所有低质量动词（做了、搞了、跟进中）
- [ ] 每条完成事项是否符合"动词 + 对象 + 结果"的公式
- [ ] 有没有"差不多"、"基本"、"还在"等模糊词

## 4. 篇幅与格式规范

### 规则

**控制在 300-500 字，使用 Markdown 格式，周五下班前发送。**

| 要求 | 标准 |
|------|------|
| 字数 | 300-500 字（太长没人读，太短信息不足） |
| 格式 | Markdown（标题/列表/加粗/表格） |
| 每条完成事项 | 不超过 2 行 |
| 发送时间 | 每周五下班前（不得拖到周末或下周一）|
| 发送对象 | 直属导师 + 团队 Leader（以团队规定为准）|

---

### ❌ 反例（格式混乱版）

```
工作汇报，本周主要做了以下事情：1.看了几篇论文，DETR和Deformable DETR，感觉挺有意思的，了解了Transformer在检测任务上的应用，以后可以参考。2.跑了实验，AP提升了。3.发现了一个数据集的问题，标注有一些错误，已经处理了。4.代码做了修改。下周继续推进，争取把效果做上去。（字数：130字，读完什么都不清楚）
```

## 5. 周报自查表（写完必过）

写完周报后，对照以下10个问题逐条检查，答"否"的立刻修改：

| # | 自查问题 | 是 | 否 |
|---|---------|----|----|
| 1 | 每条完成事项是否有具体结果，而非过程描述？| | |
| 2 | 是否有数字/指标支撑关键结论？| | |
| 3 | 问题是否写了解决方案或当前状态，而非只是罗列困难？| | |
| 4 | 下周计划是否有截止时间，且可验收？| | |
| 5 | 有没有使用"差不多/基本/还在/跟进中"等模糊词？| | |
| 6 | 格式是否整洁，有无明显错别字？| | |
| 7 | 篇幅是否在 300-500 字之间？| | |
| 8 | 是否在周五下班前准备好发送？| | |
| 9 | 需要上级支持的事项是否写清楚了诉求和时间要求？| | |
| 10 | 上周计划与本周完成是否能对上（每条都有交代）？| | |

**通过标准：** 10 题全部回答"是"才能发送。

## 附：一份完整的优秀周报样本

> 以下是一份经过脱敏处理的真实优秀周报样本，供参考。

```markdown
**周报 | 2024-04-12 | 李四**
**项目：工业缺陷检测 v2 | 汇报对象：王导师**

---

**① 本周完成事项**

1. 完成定制化 Mosaic 数据增强方案，在验证集上小目标 Recall 从 74% 提升至 79%（↑5%），
   距目标 80% 仅差 1%；代码已合并至 dev 分支（PR #27）

2. 完成 500 张产线数据标注质检（与标注团队协作），剔除质量不合格样本 42 张，
   最终可用 458 张，已导入训练流水线

3. 在产线数据上完成微调实验（3个 epoch），Recall 从 66%（直接推理）
   提升至 77%（微调后），Precision 保持在 83%

4. 输出本阶段实验总结文档（见 docs/experiments/phase2_summary.md），
   记录了全部28次实验的配置和结果，供团队参考

---

**② 关键数据**

微调前后对比（产线测试集，458张）：

| 方案 | Recall | Precision | F1 |
|------|--------|-----------|-----|
| 直接推理（无微调）| 66% | 87% | 75.1% |
| 微调 3 epoch | **77%** | **83%** | **79.9%** |
| 微调 5 epoch（正在跑）| - | - | - |

---

**③ 遇到的问题及解决方案**

- 问题：Mosaic 增强在目标尺寸 < 8px 的极小目标上引入了边界截断，
        导致约 3% 的样本标注失效
  解决：增加最小目标尺寸过滤（< 8px 的目标不参与 Mosaic 拼接），
        实测无明显性能损失，问题已解决

- 风险：微调 5 epoch 实验结果尚未出来，若 Recall 无法突破 80%，
        M3 里程碑可能需要方案B（集成 DINO 离线精检）

---

**④ 下周计划**

1. 【高】获取 5 epoch 微调结果并决策：若≥80% 则冻结方案，准备 M3 部署；
         若不足则启动方案B（周二前得出结论）
2. 【高】开始 TensorRT 推理引擎适配，目标 FPS 从 35 提升至 50+（截止周四）
3. 【中】准备 M3 里程碑汇报 PPT（截止周五）

无。本周所有阻塞均已自行解决。
```
