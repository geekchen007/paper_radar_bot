# 第四章：项目进展汇报规范

> 汇报进展不是"让上级知道你很忙"，而是"让团队对项目状态有准确的认知，能及时做出决策"。

---

## 1. 汇报节奏与时机

### 规则

**不同场景用不同的汇报方式。** 搞清楚当前是哪种场景，再决定怎么汇报。

| 场景 | 频率 | 时长 | 形式 | 核心目的 |
|------|------|------|------|---------|
| 例行周会进展同步 | 每周 | 3-5 分钟 | 口头 + 一页要点 | 让团队知道你在做什么、是否正常 |
| 里程碑节点汇报 | 按项目计划 | 15-30 分钟 | PPT | 展示阶段成果，获得决策支持 |
| 紧急风险上报 | 随时 | 5 分钟内 | 即时消息 + 口头 | 让相关人在第一时间知道并介入 |

**最重要的原则：不要等到 deadline 前才暴露问题。** 风险出现后24小时内必须主动上报，哪怕只是"我遇到了一个问题，还在排查"。

---

### ❌ 反例

```
错误的汇报节奏（真实场景还原）：

周一：开始做实验，发现数据集标注质量有问题
周二：继续排查，问题比预想的严重
周三：尝试了几个方法，都不太好用
周四：还在搞，有点担心但觉得周五能搞定
周五（周会上）：

上级：本周进度怎么样？
新人：呃……我遇到了一个数据集的问题，
     导致实验没有跑完……

上级（内心）：这个问题周一就有了，为什么现在才说？
             我本来可以在周二就帮你找其他人处理的。
```

**为什么不好：** 主动上报风险是争取资源和支持的最佳时机，拖延上报会让问题累积，最终影响里程碑。

---

### ✅ 正例

```
正确的汇报节奏（同样的情景）：

周一下午：发现数据标注问题后，立刻发消息给导师：

"[风险预警] 我在数据预处理时发现约 12% 的标注存在错误
（边界框偏移超过 10 像素），主要集中在类别 'scratch' 中。
目前尝试了自动过滤，但会损失 20% 的训练数据。
这可能影响本周的实验计划，预计 deadline 会推迟 2-3 天。

已尝试方案：
1. 阈值过滤（损失数据太多，AP 下降 3%）
2. 半自动重标注（需要 2 天时间）

需要确认：是否有已清洗过的标注版本？或者是否需要申请
标注资源重新标注这部分数据？"

结果：导师当天回复"有一个v2版本的标注，路径在XXX"
问题1天内解决，实验正常推进。
```

---

### ✏️ Checklist

- [ ] 遇到阻塞或风险是否在 24 小时内主动上报
- [ ] 是否区分了"例行同步"和"里程碑汇报"的场景
- [ ] 上报时是否带上了"已尝试的方案"和"需要什么支持"

---

## 2. 进展汇报的标准结构

### 规则

**无论哪种汇报场景，内容都要覆盖这四段。** 缺少任何一段，汇报就是不完整的。

```
① 上期目标完成情况（对齐上次承诺）
② 本期核心进展（量化结果优先）
③ 遇到的问题 / 风险（不能只报喜）
④ 下期计划（明确、可验收）
```

---

### ❌ 反例

```markdown
本周进展汇报（错误示范）

本周主要做了以下工作：
- 看了几篇相关论文
- 跑了一些实验
- 和同学讨论了一下方案
- 代码做了一些修改

下周继续推进。
```

**逐条分析问题：**

- "看了几篇相关论文" → 看了几篇？看了什么？得出什么结论？
- "跑了一些实验" → 什么实验？结果是什么？比 Baseline 好还是差？
- "和同学讨论了一下方案" → 讨论出什么结论？
- "代码做了一些修改" → 改了什么？解决了什么问题？
- "下周继续推进" → 推进什么？怎么算完成？

这份汇报让上级完全无法判断项目是否在正常推进。

---

### ✅ 正例

```markdown
本周进展汇报（2024-03-18，张三）

**① 上周目标完成情况**
- [x] 完成 FPN Neck 集成到训练框架（已合并到 dev 分支）
- [x] 完成 COCO val2017 基准实验（见②）
- [ ] 小目标数据增强方案选型（延期，原因见③）

**② 本周核心进展**

完成 FPN + 注意力机制消融实验，结果如下：

| 配置 | AP | APₛ（小目标）| FPS |
|------|-----|------------|-----|
| Baseline（YOLOv8-m）| 50.2 | 21.3 | 80 |
| + FPN Neck | 52.7 | 26.8 | 42 |
| + FPN + CBAM 注意力 | **53.9** | **28.1** | 35 |

结论：FPN 对小目标提升最显著（+5.5 AP），注意力机制进一步提升 1.3 AP，
但 FPS 从 80 降至 35，仍满足 ≥30 FPS 的需求。

**③ 遇到的问题 / 风险**

小目标数据增强方案选型延期：
- 原因：Mosaic 增强在我们的数据集上引入了大量无效拼接，
        小目标比例反而下降 12%，需要定制化修改
- 已尝试：标准 Mosaic（失败），调整 tile 尺寸（部分改善）
- 影响：增强方案选型推迟到下周三（不影响整体里程碑，有1天余量）
- 需要支持：无，自行解决中

**④ 下周计划**
1. 完成小目标定制化 Mosaic 增强方案，验证有效性（截止：周三）
2. 在真实产线数据集（500张）上验证当前最优模型（截止：周五）
3. 如产线验证结果 AP > 55%，提交里程碑汇报申请
```

---

### ✏️ Checklist

- [ ] 是否对齐了上次的目标承诺（逐条确认完成/未完成）
- [ ] 核心进展是否有量化数字（不是"效果不错"）
- [ ] 是否写了遇到的问题（不能只报喜）
- [ ] 问题是否包含影响范围和已尝试的解决方案
- [ ] 下周计划是否有明确的截止时间（不是"下周完成"）

## 3. 量化表达规范

### 规则

**进展描述必须可度量。** 遇到这些词，立刻替换：

| 禁用词 | 替换方案 |
|--------|---------|
| 差不多完成了 | 完成了 X/Y 个子任务，剩余 Z 个 |
| 基本跑通了 | 在 [数据集] 上跑通，loss 正常下降，AP 达到 XX |
| 有进展 | 具体进展：[做了什么] → [得到什么结果] |
| 还在推进 | 当前进度：[具体步骤]，预计 [日期] 完成 |
| 效果还不错 | AP 从 XX 提升至 XX，提升 XX%，超过 Baseline |
| 速度有提升 | 推理时间从 42ms 降至 28ms，提速 33% |

---

### ❌ 反例

```
"模型训练基本跑通了，效果还不错，速度也还可以，
 数据处理那边差不多完成了，下周应该能继续往前推进。"

（读完这段话，没有获取到任何有效信息）
```

---

### ✅ 正例

```
"在 COCO val2017 上，mAP@0.5:0.95 达到 53.9，
比 Baseline YOLOv8-m 提升 3.7 个百分点（50.2→53.9）。
推理速度为 35 FPS，满足 ≥30 FPS 的需求。
数据预处理流水线已完成，处理速度 120 张/秒，
覆盖了 train/val/test 三个 split，共 23,847 张图像。
下周计划完成真实产线数据集（500张）的评估，
截止时间周五下午6点。"
```

---

### ✏️ Checklist

- [ ] 是否删除了所有模糊词（差不多、基本、还不错、有进展）
- [ ] 每个进展是否都有具体数字
- [ ] 数字是否有对比基准（跟什么比，提升了多少）
- [ ] 时间节点是否精确到具体日期

---

## 4. 风险与阻塞的上报规范

### 规则

**上报风险时，必须包含三要素：**

1. **问题是什么**（现象 + 影响范围）
2. **已尝试的解决方案**（说明你认真想过）
3. **需要什么支持 / 影响哪个 deadline**（让上级知道该怎么帮你）

**上报的格式模板（可直接复制使用）：**

---

### ❌ 反例

```
消息："服务器环境有问题，我不知道怎么办，实验跑不了"

问题：
- 不知道是什么问题（什么报错？影响哪个实验？）
- 不知道你有没有自己尝试解决
- 不知道影响多大（这个里程碑受影响吗？）
- 不知道你需要什么帮助
```

---

### ✅ 正例（直接可用的上报模板）

```markdown
**[风险上报] 实验环境 CUDA 版本冲突**

**时间**：2024-03-19 14:30

**问题现象**：
A100-1 节点安装 torch==2.2.0 后与系统 CUDA 11.7 存在版本冲突，
运行 `python train.py` 时报错：
`RuntimeError: CUDA error: no kernel image is available for execution on the device`

**影响范围**：
- A100-1 节点不可用
- 原计划本周在该节点上跑的 3 组消融实验无法执行
- 如无法解决，里程碑 M2（本周五提交实验结果）将延迟 2-3 天

**已尝试的方案**：
1. 降级 torch 至 2.0.1 → 同样报错
2. 切换至 conda 新环境重装 → 问题复现
3. 查看文档，怀疑需要 CUDA 11.8+ 才能支持 torch 2.x

**需要支持**：
- 能否申请 A100-2 节点（CUDA 12.1）临时使用？
- 或者是否有已配置好环境的 Docker 镜像可以直接用？

**当前状态**：实验暂停，等待回复后继续。
```

---

### ✏️ Checklist

- [ ] 是否写清楚了具体的报错信息或现象（不是"有问题"）
- [ ] 是否说明了影响的里程碑和延迟天数
- [ ] 是否列出了已尝试的方案（至少2个）
- [ ] 是否明确写出了需要什么支持
- [ ] 是否在问题发现后 24 小时内发送了上报

---

## 5. 里程碑汇报 PPT 模板

### 规则

重要节点汇报使用固定结构，确保不遗漏关键信息。

**模板结构（7页）：**

---

```markdown
# [项目名称] 里程碑汇报
## M2：小目标检测精度提升阶段

汇报人：张三
日期：2024-03-22
汇报对象：技术 Leader + 产品 Owner

---

## 页1：项目目标回顾（1句话）

目标：在工业流水线缺陷检测任务中，将小目标（≤5mm 缺陷）的
Recall 从 61% 提升至 80%，同时保持推理速度 ≥30 FPS。

---

## 页2：里程碑时间线

M1 ✅ 2024-02-28  基础框架搭建完成，Baseline 跑通
M2 ✅ 2024-03-22  小目标检测精度提升阶段完成  ← 当前
M3    2024-04-15  产线部署与压测
M4    2024-05-01  项目验收

---

## 页3：本阶段核心成果

实验结果对比表（与 Baseline 对比）：

| 方法 | Recall（小目标）| Precision | FPS | 是否满足目标 |
|------|--------------|-----------|-----|------------|
| Baseline（YOLOv8-m）| 61% | 88% | 80 | ❌ Recall 不足 |
| 我们的方法（FPN+CBAM）| **74%** | **85%** | **35** | ⚠️ 距目标差6% |

当前状态：Recall 从 61% 提升至 74%（+13%），距目标 80% 还差 6%。

---

## 页4：关键技术发现

1. FPN Neck 是小目标提升最关键的组件（贡献 +5.5% Recall）
2. CBAM 注意力机制进一步提升（+1.3% Recall），FPS 代价可接受
3. 数据增强（定制化 Mosaic）可能贡献额外 3-5%（下阶段验证）

---

## 页5：问题与应对措施

| 问题 | 影响 | 应对措施 | 状态 |
|------|------|---------|------|
| 真实产线数据分布偏移 | Recall 下降 8% | 收集产线数据微调 | 进行中 |
| 距目标 Recall 差 6% | M3 进度有风险 | 数据增强 + 迁移学习 | 计划中 |

---

## 页6：下阶段计划与资源需求

计划：
- 4周内通过数据增强 + 在产线数据上微调，使 Recall 达到 80%
- 完成 TensorRT 推理优化，FPS 从 35 提升至 50+

资源需求：
- 申请额外 500 张产线标注数据（需要标注团队支持）
- A100 节点使用权延长至 4月底

---

## 页7：需要的决策 / 支持

请确认以下两件事：
1. 当前 74% Recall 是否可以先推进 M3（产线部署）并行优化？
2. 标注资源能否在本周内落实？（影响 M3 时间线）
```

---

### ✏️ Checklist

- [ ] 是否回顾了项目目标（让听众重新对齐）
- [ ] 时间线是否标注了当前所在位置
- [ ] 核心成果是否有数字，是否与目标对比
- [ ] 是否诚实展示了"距目标的差距"
- [ ] 是否明确写出了"需要听众做的决策"
- [ ] 资源需求是否具体（几个人、什么时候需要）
