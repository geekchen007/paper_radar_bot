# 第二章：文献调研方法

> 调研不是"看了多少篇论文"，而是"能不能用一张图说清楚这个领域的脉络"。

---

## 1. 检索规范

### 规则

**明确检索词策略，在正确的平台上找正确的论文。** 不要从中文博客开始，要从原始论文开始。

**推荐平台（按使用顺序）：**

| 平台 | 用途 |
|------|------|
| [Papers With Code](https://paperswithcode.com) | 找领域 SOTA，直接看 benchmark 排行榜 |
| [Semantic Scholar](https://www.semanticscholar.org) | 按引用量排序，找高影响力论文 |
| [arXiv](https://arxiv.org) | 找最新预印本（CV→cs.CV，NLP→cs.CL） |
| [Google Scholar](https://scholar.google.com) | 找某作者的全部论文，或引用了某篇论文的后续工作 |
| [Connected Papers](https://www.connectedpapers.com) | 可视化论文引用关系，找相关工作 |

**检索词策略：**

1. 用英文关键词，不用中文
2. 控制时间范围（例如 `2021-2024`），优先看近三年
3. 先找"综述论文"（survey / review），快速建立领域地图
4. 从综述的参考文献里找关键原始论文，从原始论文的引用里找后续进展

---

### ❌ 反例

```
调研步骤（错误示范）：

1. 百度搜索"目标检测算法"
2. 看了三篇 CSDN 博客
3. 跑去看 YOLO 的 GitHub README
4. 向导师汇报："我了解了目标检测，主要方法有 YOLO、SSD、Faster RCNN"

问题：
- 中文博客的内容可能是错的，或者已经过时
- 没有读过任何一篇原始论文，无法判断技术细节的准确性
- 缺少领域的发展脉络，不知道"为什么从 Faster RCNN 演进到 YOLO"
- 无法回答"最新的 SOTA 是什么"
```

---

### ✅ 正例

```
调研步骤（正确示范）：以"目标检测"为例

第一步：找领域地图
→ 在 Papers With Code 搜索 "object detection"
→ 找到 COCO benchmark 排行榜，记下前5名方法的名字和发表年份
→ 在 arXiv 搜索 "object detection survey 2023"，找一篇近期综述
→ 花30分钟浏览综述的第2-3节（方法分类），在脑子里建立"两阶段 vs 单阶段 vs Transformer-based"的框架

第二步：确定必读论文清单（≤10篇）
从综述参考文献 + Papers With Code 排行榜交叉确认，得到：
- Faster RCNN（两阶段经典）
- YOLO v3（单阶段经典）
- DETR（Transformer 开山之作）
- DINO（当前 SOTA 之一）
- Co-DETR（最新进展）

第三步：精读（见下一节）
```

---

### ✏️ Checklist

- [ ] 检索是否从英文关键词开始，而非中文博客
- [ ] 是否使用了 Papers With Code 找到了领域 benchmark
- [ ] 是否找到了至少一篇近三年的综述论文
- [ ] 必读论文清单是否控制在 10 篇以内（先宽后精）
- [ ] 没有把 GitHub README 或 CSDN 博客当作一手资料

## 2. 精读框架（三遍读法）

### 规则

**读论文分三遍，每遍有明确目的和时间限制。** 不要从第一页读到最后一页，那是在浪费时间。

| 遍数 | 目的 | 时间 | 阅读内容 |
|------|------|------|---------|
| 第一遍 | 判断值不值得精读 | 5-10 分钟 | 标题、摘要、图1（方法示意图）、结论第一段、实验结果表格 |
| 第二遍 | 理解核心贡献 | 20-40 分钟 | 引言（重点看"we propose"附近）、方法部分（可跳公式）、实验部分（看消融实验） |
| 第三遍 | 能复现为止 | 1-3 小时 | 方法部分（逐公式）、实验细节、附录、参考代码 |

**何时进行第三遍？** 当这篇论文是你要实现的方法，或者你需要在汇报中详细讲解时。否则第二遍已经足够。

---

### ❌ 反例

```
错误的读法：

- 从摘要开始，逐段往下读，读到一半读不懂就放弃
- 跳过实验部分，"以后再看"
- 读完了没有任何笔记，靠记忆记住内容
- 对所有论文都做第三遍精读，每天读不完两篇
- 只看中文翻译版，遇到专业术语一脸茫然

典型现象：
"我读了这篇论文，大概就是用 Transformer 做检测，效果挺好的。"
（无法说出具体的技术贡献是什么，实验提升了多少）
```

---

### ✅ 正例

以精读 **DETR**（End-to-End Object Detection with Transformers）为例：

```
第一遍（8分钟）：
- 标题：End-to-End → 关键词，意味着去掉了 NMS 等后处理
- 摘要："we present a new method that views object detection
         as a direct set prediction problem"
  → 核心贡献：把检测问题转化为集合预测，用 Transformer 直接输出结果
- 图1：Encoder-Decoder 结构 + object queries
  → 大概理解了架构
- 结论：COCO 上与 Faster RCNN 相当，训练慢
- 判断：值得精读（因为这是 Transformer 用于检测的开山之作）

第二遍（35分钟）：
- 引言第3段："Unlike existing methods, DETR does not need any
              customized layers and thus can be reproduced easily"
  → 理解了动机：现有方法依赖手工设计的组件（anchor、NMS）
- 方法部分（跳过公式细节）：
  → Encoder 处理图像特征，Decoder 通过 object queries 并行预测所有目标
  → 匈牙利算法用于训练时的目标匹配（解决了集合预测的配对问题）
- 消融实验表3：去掉 Encoder 掉 1.3 AP，去掉 Decoder attention 掉 3.9 AP
  → Decoder 的 cross-attention 是关键组件

第三遍（如需复现时）：
- 仔细看 Section 3.1 的公式，理解二分图匹配损失的具体形式
- 看附录 A 的训练细节（500 epoch！）
- 对照官方代码 github.com/facebookresearch/detr 验证理解
```

---

### ✏️ Checklist

- [ ] 第一遍是否在 10 分钟内完成并做出"值不值得精读"的判断
- [ ] 第二遍是否重点看了引言的"we propose"和消融实验
- [ ] 是否只对真正需要的论文做了第三遍精读
- [ ] 读完后是否立即填写了笔记模板（见下一节）

## 3. 论文笔记模板

### 规则

**每篇精读的论文必须输出一份结构化笔记，当天写完。** 不要依赖记忆，一周后你会什么都不记得。

---

### ❌ 反例

```markdown
# DETR 笔记

这篇论文讲了用 Transformer 做目标检测，效果和 Faster RCNN 差不多，
但是训练需要很多 epoch。创新点是用了 object query 和匈牙利算法。
以后有机会复现一下。

参考：https://arxiv.org/abs/2005.12872
```

**为什么不好：** 没有具体数字，没有局限性分析，没有与自己工作的关联，两个月后完全无法利用这份笔记。

---

### ✅ 正例（使用标准模板）

```markdown
# 论文笔记：DETR

## 基本信息
- **标题**：End-to-End Object Detection with Transformers
- **作者**：Nicolas Carion et al.（Facebook AI）
- **来源**：ECCV 2020
- **arXiv**：https://arxiv.org/abs/2005.12872
- **代码**：https://github.com/facebookresearch/detr
- **阅读日期**：2024-03-15
- **阅读程度**：第二遍（理解核心贡献）

---

## 解决什么问题

目标检测方法（如 Faster RCNN）依赖大量手工设计的组件：anchor 生成、
NMS 后处理、RPN 等。这些组件需要专家知识调参，且难以端到端优化。

---

## 核心方法（3句话）

1. 将检测问题转化为**集合预测**问题，Transformer Decoder 直接输出
   N 个预测框（N 固定，如 100），无需 NMS。
2. 训练时用**匈牙利算法**做预测框与 GT 的一对一最优匹配，
   避免重复预测同一目标。
3. Encoder 提取全局图像特征，Decoder 通过可学习的 object queries
   并行预测所有目标的类别和位置。

---

## 关键实验结论

| 指标 | DETR | Faster RCNN（对比基线） |
|------|------|------------------------|
| COCO AP | 42.0 | 40.2 |
| 训练 epoch | 500 | ~12 |
| 大目标 AP | **61.1** | 52.9（DETR 显著更好）|
| 小目标 AP | 20.5 | **23.7**（DETR 更差）|

**局限性：** 训练极慢（500 epoch vs 12 epoch）；小目标检测差（全局注意力无法聚焦局部）。

---

## 与我工作的关联

- 我们的任务是工业缺陷检测（小目标为主）→ DETR 原版不适合，
  但可以参考其 set prediction 思路
- 后续跟进 Deformable DETR（解决了训练慢和小目标的问题）

---

## 待跟进的论文

- [ ] Deformable DETR（ICLR 2021）— 解决收敛慢和小目标问题
- [ ] Conditional DETR（ICCV 2021）— 加速训练收敛
- [ ] DINO（ICLR 2023）— 目前 DETR 系列最强
```

---

### ✏️ Checklist

- [ ] 是否填写了所有基本信息（含代码链接）
- [ ] "解决什么问题"是否说清楚了现有方法的不足
- [ ] "核心方法"是否控制在 3 句话以内
- [ ] 实验结论是否有具体数字（不是"效果很好"）
- [ ] 是否写了局限性（任何方法都有局限性）
- [ ] 是否写了"与我工作的关联"（否则调研脱离实际）
- [ ] 是否列出了待跟进的论文

## 4. 综述报告写法

### 规则

**调研结束后需要产出一份综述报告，供团队共享。** 不能只在脑子里"大概了解了一下"，要形成可传递的知识资产。

综述报告的标准结构：

```
1. 背景与问题定义（半页）
2. 主流方法分类与对比表（核心）
3. 各方法优劣分析
4. 推荐方向（结合我们的具体任务）
5. 参考文献列表
```

---

### ❌ 反例

```
口头汇报："我调研了目标检测领域，看了几篇论文，感觉 DETR 比较好，
           建议我们用 DETR。"

问题：
- 没有说清楚"比较好"的依据是什么数据
- 没有考虑我们任务的具体需求（小目标？实时性要求？）
- 团队其他人无法从中获取信息，知识不可传递
- 两个月后这份"调研"就消失了
```

---

### ✅ 正例（综述报告样例）

```markdown
# 目标检测方法综述报告

**调研人**：张三  
**日期**：2024-03-20  
**调研背景**：为工业流水线缺陷检测任务选型，需支持小目标、推理速度 ≥ 30 FPS

---

## 1. 背景与问题定义

目标检测任务要求在图像中定位并识别所有目标，输出每个目标的
边界框（bounding box）和类别标签。核心挑战包括：多尺度目标、
遮挡、实时性要求。

当前主流方法按架构分为三类：
- **两阶段方法**：先生成候选区域，再分类（精度高，速度慢）
- **单阶段方法**：直接预测（速度快，小目标弱）
- **Transformer-based**：全局注意力（精度高，训练慢）

---

## 2. 主流方法对比

| 方法 | 类型 | COCO AP | 速度（FPS） | 小目标 AP | 优点 | 缺点 |
|------|------|---------|------------|----------|------|------|
| Faster RCNN | 两阶段 | 40.2 | 5 | 23.7 | 精度稳定 | 速度慢 |
| YOLOv8-m | 单阶段 | 50.2 | 80 | 32.1 | 速度快 | 小目标一般 |
| DETR | Transformer | 42.0 | 28 | 20.5 | 端到端简洁 | 训练慢，小目标差 |
| Deformable DETR | Transformer | 46.2 | 27 | 29.6 | 兼顾精度速度 | 复杂度高 |
| DINO | Transformer | 58.5 | 20 | 42.1 | 最高精度 | 推理较慢 |

---

## 3. 各方法优劣分析

（针对我们的任务：小目标缺陷检测，需要 ≥ 30 FPS）

- **Faster RCNN**：精度够，但 5 FPS 不满足实时性要求，排除
- **YOLOv8**：速度满足，小目标 AP 32.1，可作为 Baseline
- **DINO**：精度最高，但 20 FPS 不满足实时性，可作为离线精检方案
- **Deformable DETR**：27 FPS 接近要求，可通过 TensorRT 优化至 35 FPS

---

## 4. 推荐方向

**短期（1个月内）**：以 YOLOv8-m 为 Baseline，快速验证数据质量和任务难度。  
**中期**：尝试 Deformable DETR + TensorRT 优化，争取在精度和速度上超越 Baseline。  
**参考工作**：RT-DETR（百度，2023）已在实时场景取得 YOLOv8 级别速度 + DINO 级别精度，重点跟进。

---

## 5. 参考文献

1. Ren et al., "Faster R-CNN", NeurIPS 2015
2. Jocher et al., "YOLOv8", Ultralytics 2023
3. Carion et al., "DETR", ECCV 2020
4. Zhu et al., "Deformable DETR", ICLR 2021
5. Zhang et al., "DINO", ICLR 2023
6. Zhao et al., "RT-DETR", CVPR 2024
```

---

### ✏️ Checklist

- [ ] 报告是否写明了调研背景和具体任务需求
- [ ] 方法对比表是否包含具体数字（AP、FPS）
- [ ] 是否分析了各方法与**我们任务**的匹配度（而非泛泛的优缺点）
- [ ] 推荐方向是否给出了明确建议（短期 + 中期）
- [ ] 参考文献是否列出了完整信息
- [ ] 报告是否上传到了团队共享的位置（Git 仓库或文档平台）
