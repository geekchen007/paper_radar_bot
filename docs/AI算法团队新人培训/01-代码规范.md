# 第一章：代码编程规范

> 好代码的标准不是"能跑"，而是"三个月后的你（或你的同事）能快速读懂并修改"。

---

## 1. 命名规范

### 规则

| 类型 | 风格 | 示例 |
|------|------|------|
| 变量、函数、方法 | `snake_case` | `learning_rate`, `load_checkpoint` |
| 类名 | `PascalCase` | `ResNetBackbone`, `TrainConfig` |
| 常量 | `UPPER_SNAKE_CASE` | `MAX_EPOCH`, `DEFAULT_BATCH_SIZE` |
| 模块/包 | 全小写，可用下划线 | `data_loader`, `utils` |
| 私有成员 | 单下划线前缀 | `_hidden_dim`, `_build_head` |

**命名的核心原则：名字本身就是注释。** 看到名字就知道这个变量/函数是做什么的。

---

### ❌ 反例

```python
# 问题1：无语义命名，三天后自己也不知道是什么
a = 0.001
tmp2 = model(x)
data1, data2 = split(dataset)

# 问题2：缩写过度，必须联系上下文才能猜
lr = 0.001       # lr 还算常见，但 ep、bs 就不直观了
ep = 100
bs = 32

# 问题3：类型信息冗余（匈牙利命名法），Python 有类型注解后无需如此
strName = "resnet"
intEpoch = 100
listImages = []

# 问题4：拼音命名，团队里有外籍成员时完全无法协作
pici_da_xiao = 32
xue_xi_lv = 0.001
```

**为什么不好：** 看到 `a`、`tmp2`、`data1` 时，读者必须回溯上下文才能理解含义，增加理解成本；拼音命名在国际团队中制造障碍。

---

### ✅ 正例

```python
# 清晰表达含义，见名知意
learning_rate = 0.001
batch_size = 32
max_epochs = 100

# 函数名是动词短语，说明"做什么"
def load_checkpoint(checkpoint_path: str) -> dict:
    ...

def compute_iou(box_a: torch.Tensor, box_b: torch.Tensor) -> float:
    ...

# 类名是名词，描述"是什么"
class EarlyStopping:
    def __init__(self, patience: int = 5):
        self.patience = patience
        self._counter = 0          # 私有属性用单下划线

# 常量全大写，放在文件顶部
MAX_EPOCHS = 200
DEFAULT_LR = 1e-3
CHECKPOINT_DIR = "outputs/checkpoints"
```

**为什么好：** 任何人读到 `learning_rate`、`compute_iou` 都能立刻理解含义，不需要查上下文。

---

### ✏️ Checklist

- [ ] 所有变量名是否有明确语义（无 `a`、`tmp`、`data1`）
- [ ] 函数名是否是动词短语
- [ ] 类名是否是名词，使用 PascalCase
- [ ] 常量是否全大写并放在文件顶部
- [ ] 没有使用拼音命名

## 2. 项目目录结构

### 规则

一个标准的 AI 算法项目目录如下，**数据、模型、配置、脚本、输出必须分层**：

```
project_name/
├── configs/               # 所有超参数配置文件（yaml/json）
│   ├── base.yaml
│   └── experiment_01.yaml
├── data/                  # 数据相关
│   ├── raw/               # 原始数据（不做任何处理）
│   ├── processed/         # 预处理后的数据
│   └── splits/            # 训练/验证/测试集划分文件
├── src/                   # 核心源代码
│   ├── datasets/          # 数据加载与增强
│   ├── models/            # 模型定义
│   ├── losses/            # 损失函数
│   ├── utils/             # 工具函数
│   └── trainer.py         # 训练主循环
├── scripts/               # 入口脚本
│   ├── train.py
│   ├── evaluate.py
│   └── infer.py
├── outputs/               # 所有实验输出（加入 .gitignore）
│   ├── checkpoints/
│   ├── logs/
│   └── visualizations/
├── notebooks/             # 探索性分析 Jupyter Notebook
├── tests/                 # 单元测试
├── requirements.txt       # 依赖清单
└── README.md              # 项目说明
```

---

### ❌ 反例

```
my_project/
├── train_final.py
├── train_final_v2.py
├── train_final_真的最终版.py      # ← 版本管理应该交给 Git
├── model.py
├── model_new.py
├── model_backup.py                # ← 备份文件不应存在于仓库
├── data.csv
├── data_processed.csv
├── test.ipynb
├── test2.ipynb
├── aaa.py                         # ← 无意义文件名
└── 0317实验结果.txt               # ← 用日期命名且中文，无法跨平台
```

**为什么不好：** 两周后自己都不知道哪个是"最终版"；新人无法快速定位入口文件；数据和代码混在一起难以管理。

---

### ✅ 正例

参考 [CLAIRE-Labo/python-ml-research-template](https://github.com/CLAIRE-Labo/python-ml-research-template) 的结构，核心原则：

```
detection_project/
├── configs/
│   ├── base.yaml              # 基础配置
│   └── ablation_fpn.yaml      # 消融实验配置（继承 base，只覆盖差异项）
├── src/
│   ├── datasets/
│   │   ├── __init__.py
│   │   ├── coco.py            # COCO 数据集加载
│   │   └── transforms.py      # 数据增强
│   ├── models/
│   │   ├── __init__.py
│   │   ├── backbone.py
│   │   ├── neck.py
│   │   └── head.py
│   └── utils/
│       ├── metrics.py
│       └── visualization.py
├── scripts/
│   ├── train.py               # 唯一的训练入口
│   └── evaluate.py
├── outputs/                   # 在 .gitignore 中排除
└── README.md                  # 写清楚：怎么安装环境、怎么跑训练
```

---

### ✏️ Checklist

- [ ] 是否有独立的 `configs/`、`src/`、`outputs/` 目录
- [ ] 是否有且只有一个训练入口脚本
- [ ] `outputs/` 是否加入了 `.gitignore`
- [ ] 是否有 `requirements.txt` 或 `pyproject.toml`
- [ ] 没有 `_v2`、`_final`、`_backup` 之类的文件名
- [ ] `README.md` 是否写清楚了如何运行

## 3. 函数与注释规范

### 规则

**单一职责原则：一个函数只做一件事。** 判断标准：函数名能否用一个动词短语描述清楚？如果需要"并且"连接，就要拆分。

所有公开函数必须有 **docstring** 和 **类型注解**。

---

### ❌ 反例

```python
# 问题1：一个函数做了5件事（加载数据、训练、验证、保存、绘图）
def run(config):
    # 加载数据
    data = pd.read_csv(config['data_path'])
    X = data.drop('label', axis=1).values
    y = data['label'].values
    X_train, X_val = X[:800], X[800:]
    y_train, y_val = y[:800], y[800:]

    # 训练
    model = MyModel(config['hidden_dim'])
    for epoch in range(config['epochs']):
        loss = model.train_step(X_train, y_train)
        print(f"epoch {epoch}, loss: {loss}")

    # 验证
    acc = model.evaluate(X_val, y_val)
    print(f"val acc: {acc}")

    # 保存
    torch.save(model.state_dict(), 'model.pth')

    # 画图
    plt.plot(losses)
    plt.savefig('loss.png')

# 问题2：无 docstring，无类型注解
def process(x, mode):
    if mode == 1:
        return x * 2
    else:
        return x + 1
```

**为什么不好：** `run()` 函数无法单独测试任何一个步骤；改数据加载逻辑时必须阅读整个函数；`process()` 的入参含义完全不清楚。

---

### ✅ 正例

```python
from typing import Tuple
import torch
import torch.nn as nn


def load_dataset(data_path: str, val_ratio: float = 0.2) -> Tuple[torch.Tensor, torch.Tensor]:
    """从 CSV 文件加载数据集并划分训练/验证集。

    Args:
        data_path: CSV 文件路径，需包含 'label' 列。
        val_ratio: 验证集比例，默认 0.2。

    Returns:
        (train_data, val_data) 元组，每项形状为 (N, feature_dim)。
    """
    data = pd.read_csv(data_path)
    split_idx = int(len(data) * (1 - val_ratio))
    return data[:split_idx], data[split_idx:]


def train_one_epoch(
    model: nn.Module,
    dataloader: torch.utils.data.DataLoader,
    optimizer: torch.optim.Optimizer,
) -> float:
    """运行一个 epoch 的训练，返回平均损失。

    Args:
        model: 待训练的模型。
        dataloader: 训练数据加载器。
        optimizer: 优化器。

    Returns:
        本 epoch 的平均损失值。
    """
    model.train()
    total_loss = 0.0
    for batch in dataloader:
        optimizer.zero_grad()
        loss = model.compute_loss(batch)
        loss.backward()
        optimizer.step()
        total_loss += loss.item()
    return total_loss / len(dataloader)


def save_checkpoint(model: nn.Module, save_path: str, epoch: int) -> None:
    """保存模型权重到指定路径。

    Args:
        model: 要保存的模型。
        save_path: 保存目录。
        epoch: 当前 epoch 数，会拼入文件名。
    """
    path = f"{save_path}/epoch_{epoch:03d}.pth"
    torch.save(model.state_dict(), path)
```

**为什么好：** 每个函数只做一件事，可以独立测试；docstring 告诉读者入参含义和返回值；类型注解让 IDE 能做自动补全和检查。

---

### ✏️ Checklist

- [ ] 每个函数是否只做一件事（函数名不需要用"并且"描述）
- [ ] 所有公开函数是否有 docstring
- [ ] 所有参数和返回值是否有类型注解
- [ ] 函数长度是否在 50 行以内（超出考虑拆分）
- [ ] 没有注释掉的死代码（`# old version:`）

## 4. Git 提交规范

### 规则

**原子提交：每个 commit 只做一件事，保证可独立回滚、可独立理解。**

Commit message 格式：

```
<type>(<scope>): <subject>

[可选 body：解释为什么这样改]
```

**type 对照表：**

| type | 含义 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `refactor` | 重构（不改变功能） |
| `perf` | 性能优化 |
| `test` | 添加或修改测试 |
| `docs` | 文档变更 |
| `chore` | 构建/依赖/配置等杂项 |

---

### ❌ 反例

```bash
# 问题1：无意义的 message，完全不知道改了什么
git commit -m "update"
git commit -m "fix bug"
git commit -m "改了一些东西"
git commit -m "aaaa"

# 问题2：一次 commit 改了所有文件
git add .
git commit -m "完成第一版"
# 结果这个 commit 里包含：新模型结构 + 数据预处理 + 配置文件 + README
# 任何一个部分出问题都无法单独回滚

# 问题3：把未完成的代码提交到主分支
git commit -m "wip: half done"
# 主分支此时跑不通
```

**为什么不好：** 三周后 `git log` 一看全是 "update"，完全无法追溯修改历史；一个大 commit 出问题只能整体回滚，无法精准定位。

---

### ✅ 正例

```bash
# 正例1：新功能，类型 + scope + 清晰的 subject
git commit -m "feat(model): add FPN neck to support multi-scale detection"

# 正例2：Bug 修复，body 解释了为什么
git commit -m "fix(dataloader): handle empty annotation files gracefully

Previously raised KeyError when annotation JSON had no 'bbox' key.
Added early return with warning log instead."

# 正例3：性能优化
git commit -m "perf(trainer): replace for-loop NMS with batched torch ops, 3x speedup"

# 正例4：配置/杂项变更
git commit -m "chore(deps): upgrade torch from 2.0.1 to 2.2.0"

# 正例5：文档
git commit -m "docs(readme): add environment setup and quickstart guide"
```

**分批提交的操作示例：**

```bash
# 先只暂存模型相关文件，精确控制提交内容
git add src/models/neck.py src/models/head.py
git commit -m "feat(model): add FPN neck and detection head"

# 再提交配置文件
git add configs/fpn_experiment.yaml
git commit -m "chore(config): add FPN experiment config"

# 最后提交文档
git add README.md
git commit -m "docs: update architecture diagram for FPN version"
```

---

### ✏️ Checklist

- [ ] commit message 是否符合 `type(scope): subject` 格式
- [ ] subject 是否用英文祈使句（不超过 72 字符）
- [ ] 没有使用 "update"、"fix bug"、"改了一些东西" 等无信息词
- [ ] 是否按逻辑拆分为多个原子 commit（没有 `git add .` 一次提交所有）
- [ ] 主分支上的代码是否随时可以跑通

## 5. 实验管理规范

### 规则

**超参数必须写在配置文件里，禁止硬编码在代码中。** 每次实验必须有记录，保证结果可复现。

---

### ❌ 反例

```python
# 问题1：超参数硬编码在代码里
model = ResNet(layers=50)
optimizer = torch.optim.SGD(model.parameters(), lr=0.001, momentum=0.9)
scheduler = StepLR(optimizer, step_size=30, gamma=0.1)

for epoch in range(200):           # ← 200 是哪来的？
    train(model, train_loader, optimizer)
    if epoch % 10 == 0:            # ← 10 是哪来的？
        save_checkpoint(model, "checkpoints/model.pth")  # ← 路径硬编码

# 问题2：改了超参数但没有记录，下周不知道哪个实验用了什么参数
# （只有一个 model.pth，覆盖了上一次的结果）
```

**为什么不好：** 复现实验时需要看代码才能知道用了什么超参数；多次实验的结果互相覆盖，无法对比。

---

### ✅ 正例

**第一步：超参数写进配置文件**

```yaml

model:
  name: "ResNet50FPN"
  backbone_layers: 50
  pretrained: true

optimizer:
  type: "SGD"
  lr: 0.001
  momentum: 0.9
  weight_decay: 0.0001

scheduler:
  type: "StepLR"
  step_size: 30
  gamma: 0.1

training:
  max_epochs: 200
  batch_size: 32
  save_interval: 10
  output_dir: "outputs/fpn_baseline"

data:
  train_path: "data/splits/train.txt"
  val_path: "data/splits/val.txt"
  num_workers: 4
```

### ✏️ Checklist

- [ ] 所有超参数是否写在 `configs/` 下的配置文件中
- [ ] 代码中是否没有硬编码的数字（学习率、epoch 数、路径等）
- [ ] 每次实验是否有独立的 `output_dir`（不会覆盖上次结果）
